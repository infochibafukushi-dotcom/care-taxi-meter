# 08_本番運用仕様

介護タクシーメーターアプリの本番運用版確定仕様。既存の要件定義・画面設計・料金計算仕様・運行状態仕様・管理画面仕様・GPS仕様よりも、本書を優先する。

## 1. 料金計算

### 1.1 障害者割引

- 実装対象とする。
- 割引対象は「基本運賃」と「時間距離併用運賃」の合計額のみ。
- 割引率は10%。
- 10円未満は切り捨てる。
- 以下は割引対象外とする。
  - 待機料金
  - 院内付き添い料金
  - 介助料金
  - 実費
  - 予約迎車料金
  - 特殊車両料金
  - 時間指定料金

### 1.2 タクシー券

- 障害者割引適用後に適用する。
- 複数枚入力できる。
- 1枚ごとに以下を保存する。
  - 金額
  - 自治体名
  - 券番号
- タクシー券利用後の請求残額が0円未満になる場合は、請求残額を0円で止める。
- タクシー券の適用対象は、障害者割引後かつその他料金加算前の請求対象額とする。

### 1.3 時間距離併用運賃

- 内訳に独立表示する。
- 表示名称は「時間距離併用運賃」で統一する。
- 表示対象は以下すべて。
  - メーター画面
  - 精算画面
  - 案件詳細画面
  - 領収書
  - 利用明細書

### 1.4 運賃設定

- 料金表は店舗単位で1つ管理する。
- 「送迎開始」ボタン押下時の料金設定をスナップショットとして案件に保存する。
- 過去案件は当時の料金設定で再現できるようにする。

### 1.5 管理画面で設定可能な料金項目

#### 基本運賃

- 初乗距離
- 初乗料金
- 加算距離
- 加算料金

#### 時間距離併用運賃

- 低速判定速度
- 時間加算単位（秒）
- 時間加算金額

#### 予約迎車料金

- 有効／無効
- 金額
- 案件単位でON/OFF可能
- 1案件1回のみ

#### 特殊車両料金

- 有効／無効
- 金額
- 案件単位でON/OFF可能
- 1案件1回のみ

#### 深夜早朝割増

- 管理画面でON/OFFを切り替える。
- デフォルトはOFF。
- 設定項目は以下。
  - 有効／無効
  - 開始時刻
  - 終了時刻
  - 割増率
- 適用対象は以下。
  - 基本運賃
  - 時間距離併用運賃
- 「送迎開始」ボタン押下時の設定をスナップショット保存する。

#### 時間指定料金

- 管理画面でON/OFFを切り替える。
- デフォルトはOFF。
- 設定項目は以下。
  - 有効／無効
  - 固定料金
  - 時間帯別料金設定
- 適用条件は「予約時間指定あり」。
- 案件単位で適用できる。
- 「送迎開始」ボタン押下時の設定をスナップショット保存する。

## 2. 運行状態・ボタン制御

### 2.1 状態遷移

標準状態遷移は以下とする。

```text
空車
↓
走行中
↓
待機中
↓
走行中
↓
院内付き添い中
↓
走行中
↓
精算前
↓
案件終了
```

### 2.2 ボタン制御

- 待機は送迎開始後のみ利用可能。
- 院内付き添いは送迎開始後のみ利用可能。
- 精算は送迎開始後のみ利用可能。
- 案件終了後は以下を追加不可とする。
  - 介助
  - 実費
  - 予約迎車
  - 特殊車両料金
- UIは状態に応じてボタンをdisabledにする。

### 2.3 修正範囲

- 精算前はドライバーが修正できる。
- 精算後はManager以上が修正できる。

### 2.4 二重課金防止

- 待機中は時間距離併用運賃を停止する。
- 院内付き添い中は時間距離併用運賃を停止する。
- 以下の同時課金は禁止する。

| 組み合わせ | 可否 |
| --- | --- |
| 低速時間加算＋待機 | 不可 |
| 低速時間加算＋院内付き添い | 不可 |
| 待機＋院内付き添い | 不可 |

- 待機中に移動した場合は、待機終了後に走行状態へ戻す。

## 3. GPS・距離計測

- GPS取得間隔は5秒固定。
- GPS精度が30mを超えるログは除外する。
- 移動距離5m未満はノイズとして除外する。
- 5秒以内に500mを超える移動は異常値として除外する。
- GPS欠損区間は課金しない。
- 距離修正は管理者のみ可能。
- GPSエラー時も案件保存は許可する。

### 3.1 低速判定

- 10km/h以下を低速扱いとする。
- 10km/hちょうども低速扱いとする。
- 速度取得不可時はGPS差分速度を利用する。
- 速度不明時は課金しない。
- 判定方式は累積時間とする。

### 13.2 案件番号フォーマット

案件番号は以下の形式とする。

```text

```

例は以下。

```text

260607-CHB01-0002

```

### 13.3 採番ルール

- 店舗単位で独立採番する。
- 日次でリセットし、毎日0001から開始する。
- 採番タイミングは「送迎開始」ボタン押下時とする。
- 領収書番号は案件番号と共通利用する。
- 店舗コードが未設定の既存店舗では、店舗IDまたは店舗名から英数字のみを抽出し、先頭5文字を店舗コードとして利用する。
- 店舗ID・店舗名から英数字を抽出できない場合は `STORE` をfallback店舗コードとする。

## 15. 事前確定M（予約連携）

### 15.1 定義

- 事前確定Mは、予約サイト（LP）で事前に確定した運賃を本体とするメーター方式である。
- `meterMode` は `fixed` とする。
- 距離・時間による運賃の再計算は行わない。
- 予約時の `confirmedFareYen` を事前確定運賃本体として扱う。
- 介助料金・実費・待機料金・院内付き添い料金・その他追加料金は、事前確定運賃本体とは別枠で加算する。
- 合計請求額は「事前確定運賃本体 + 追加料金（割引・タクシー券・支払内訳は既存仕様どおり）」とする。

### 15.2 業務フロー

1. LPで見積を作成し、利用者が確定運賃と見積内容に同意する。
2. 同意内容と見積スナップショットを `reservation-v4` に保存する（`quote_consents` / `quote_snapshot` / `snapshotHash`）。
3. 利用者が事前確定運賃付きで予約する。
4. メーターアプリ（GitHub Pages PWA）が `reservation-v4` から予約一覧・予約詳細を読み取る。
5. 予約詳細で整合性検証を確認し、「事前確定Mで開始」を実行する（`start-fixed-fare`）。
6. 車両選択後、fixed CasePage へ遷移し、固定運賃で運行を開始する。
7. 走行・運行終了後、精算画面（SettlementPanel）で精算を保存する（`caseRecords`）。
8. 領収書・レシートを任意で発行する。運賃本体は「事前確定運賃」として表示する。
9. `caseRecords` 保存成功後、`complete-fixed-fare` を呼び出し、予約のメーター状態を完了とする。
10. 予約詳細へ戻り、「事前確定M 完了」を表示する。

### 15.3 本番構成

```text
[GitHub Pages PWA]
  https://infochibafukushi-dotcom.github.io/care-taxi-meter/
        │
        │ fetch（CORS: ALLOWED_ORIGIN のみ）
        ▼
[Cloudflare Worker: driver-proxy]
  https://care-taxi-meter-driver-proxy.info-chibafukushi.workers.dev
  - METER_DRIVER_TOKEN を Bearer として付与
  - 許可 API: GET/POST /api/driver/reservations 系のみ
        │
        │ Service Binding / RESERVATION_V4_ORIGIN
        ▼
[reservation-v4 本番 Worker]
  - 予約一覧・詳細・start-fixed-fare・complete-fixed-fare
  - meter_fixed_fare_runs テーブル
        │
        ▼
[Firebase / Firestore]
  - caseRecords（案件・精算・監査）
```

- フロントは `VITE_RESERVATION_API_BASE_URL` に driver-proxy の公開 URL のみを設定する。
- `METER_DRIVER_TOKEN` は Cloudflare Worker Secret のみに保持し、フロント・GitHub・`.env` コミット・`dist` ビルド成果物には含めない。
- driver-proxy は現時点では「トークン非露出プロキシ」である。CORS 制限だけでは curl 等の直接アクセスを完全には防げない。
- 次フェーズで Firebase ID Token 検証を driver-proxy に追加し、認可を強化する予定である。

### 15.4 整合性検証

予約詳細画面および driver API 応答で、以下を検証する。

| 項目 | 内容 |
| --- | --- |
| `snapshotHashVerified` | 保存済み `snapshotHash` と見積スナップショットの再計算結果が一致すること |
| `confirmedFareMatchesSnapshot` | 予約の `confirmedFareYen` がスナップショット上の確定運賃と一致すること |
| 同意スナップショット | 同意記録の `snapshotHash` が予約のスナップショットと一致すること（`consentSnapshotHashMatches`） |

整合性検証に失敗した予約は、運行開始前に要確認とする。

### 15.5 監査保存項目

#### caseRecords（Firebase / Firestore）

fixed 案件の精算保存時に、以下を案件データへ保存する。

```text
meterMode: fixed
reservationId
confirmedFareYen
snapshotHash
```

上記に加え、精算金額・支払内訳・介助・実費等は既存の caseRecords 項目どおり保存する。

#### meter_fixed_fare_runs（reservation-v4 / D1）

| 操作 | API | 保存内容 |
| --- | --- | --- |
| 運行開始 | `POST .../start-fixed-fare` | `status: in_progress`, `started_at`, `confirmed_fare_yen`, `snapshot_hash` |
| 運行完了 | `POST .../complete-fixed-fare` | `status: completed`, `completed_at` |

`complete-fixed-fare` は `caseRecords` 保存成功後にのみ呼び出す。API 失敗時は再試行可能な状態を維持する。

### 15.6 領収書・レシート表示

- fixed 案件の運賃本体ラベルは「事前確定運賃」とする。
- 表示金額は `confirmedFareYen`（予約時確定運賃）を用いる。
- 介助料金・実費・待機料金・院内付き添い料金・その他追加料金は別行で表示し、事前確定運賃本体に混ぜない。
- 利用明細書（`statementPdf`）は現時点で fixed 専用表示に未対応である。事前確定M案件の運賃本体の公式表示は A4領収書・サーマルレシートを用いる。
